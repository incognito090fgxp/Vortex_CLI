# Vortex CLI Module Workflow Guide (Updated)

Эта инструкция обязательна для выполнения при добавлении, перемещении или удалении любых компонентов системы.

## 1. Регистрация модуля
Каждый новый файл или папка должны быть зафиксированы в `vortex/registry.py`.
- Добавьте путь к файлу в секцию `# --- ПУТИ К ФАЙЛАМ ---`.
- Добавьте строковое имя модуля в секцию `# --- РЕЕСТР МОДУЛЕЙ ---`.
- Все пути должны формироваться через `os.path.join` для кросс-платформенной совместимости.

## 2. Обновление документации архитектуры
После внесения изменений ОБЯЗАТЕЛЬНО обновите `vortex/info.txt`.
- Опишите ответственность нового модуля.
- Если модуль сложный (как UI Engine), создайте отдельный `README.md` внутри папки модуля.
- **GEMINI.md**: Обновите системную карту файлов, если изменились корневые компоненты.

## 3. Загрузчики и Точки Входа
- **Корневые загрузчики**: `vortex.bat` (Windows) и `vortex.run` (Unix/Git Bash) являются основными точками входа. Они автоматически определяют `venv` в корне проекта.
- **Entry Point**: Пакетная точка входа определена в `vortex/__init__.py` как `main()`, проксирующая на `vortex.core.cli:main`.
- **pyproject.toml**: Поддерживайте `project.scripts` синхронизированным с главной точкой входа.

## 4. Стандарты импорта и Кросс-платформенность
- Используйте относительные импорты внутри пакета (напр. `from ..ui.engine import pager`).
- Все глобальные константы (пути, версии) берите из `vortex.config.manager` или `vortex.registry`.
- **Зависимости**: Используйте платформенные маркеры в `pyproject.toml` (например, `; sys_platform == 'win32'`) для бинарных пакетов, не поддерживаемых в Termux.
- **UpdateManager**: При добавлении новых маркеров обновляйте логику `_should_check_dep` в `updater.py`.

## 5. UI компоненты
- Вся интерактивная работа с таблицами и списками ДОЛЖНА проходить через `vortex.ui.engine`.
- Бизнес-логика (Git, SQL, Auth) не должна содержать кода отрисовки панелей или таблиц.

## 6. Проверка
Перед завершением задачи убедитесь, что:
1. Модуль есть в `registry.py`.
2. Модуль описан в `info.txt`.
3. Модуль следует принципам "худой логики" (интерфейс отделен от данных).
4. Проверена совместимость изменений с Windows и Termux (Android).
